## 五、依赖解析与依赖图构建设计

系统支持 Java、JavaScript、Python 三类项目配置文件：

```text
Java：pom.xml
JavaScript：package.json
Python：requirements.txt
```

解析后统一封装为依赖组件对象，主要字段包括：

```text
组件名称
组件版本
语言类型
所属服务
依赖类型
来源文件
命名空间
```

依赖图使用有向图建模，包含两类节点：

```text
服务节点：表示项目中的模块或服务
组件节点：表示具体依赖库
```

边包括：

```text
服务节点 -> 组件节点：表示服务依赖某组件
服务节点 -> 服务节点：表示服务之间的调用或影响关系
```

因此，系统构建的不是单一语言的依赖树，而是“服务模块 + 跨语言依赖组件 + 服务调用关系”的统一依赖关系图。

## 六、漏洞版本匹配算法设计

漏洞匹配模块需要判断项目依赖组件的当前版本是否落入漏洞数据库给出的受影响版本范围。最初可以通过提取版本号中的数字并转换为元组进行比较，例如：

```text
2.14.1 -> (2, 14, 1)
2.17.0 -> (2, 17, 0)
```

若漏洞影响范围为：

```text
version < 2.17.0
```

则版本 `2.14.1` 会被判定为受影响版本。

但是实际软件生态中版本号并不总是纯数字形式，常见情况包括：

```text
1.0.0-beta
1.0.0-rc.1
2.0.0.Final
1.2.3-SNAPSHOT
^4.17.20
~1.3.0
1.2.x
~=1.4.2
```

因此，系统引入面向多语言生态的版本约束匹配算法。该算法首先对组件版本号进行规范化处理，去除版本号前缀中的 `^`、`~`、`>=`、`<=` 等符号，并识别版本中的预发布标识。随后，系统根据不同约束类型将版本范围转换为统一的比较表达式。

示例：

```text
^4.17.20 -> version >= 4.17.20 且 version < 5.0.0
~1.3.0 -> version >= 1.3.0 且 version < 1.4.0
~=1.4.2 -> version >= 1.4.2 且 version < 1.5.0
1.2.x -> version >= 1.2.0 且 version < 1.3.0
```

预发布版本和 Maven 常见后缀处理示例：

```text
1.0.0-rc.1 < 1.0.0
1.0.0-SNAPSHOT < 1.0.0
1.0.0.Final ≈ 1.0.0
```

最终匹配规则可表示为：

```text
Match(c_i, v_j) = 1，当组件名称相同、语言类型相同且组件版本属于漏洞影响范围；
否则 Match(c_i, v_j) = 0。
```

或写成公式：

\[
Match(c_i,v_j)=
\begin{cases}
1, & name(c_i)=name(v_j) \land lang(c_i)=lang(v_j) \land version(c_i)\in Range(v_j)\\
0, & otherwise
\end{cases}
\]

## 七、漏洞传播路径分析与风险评估公式

系统基于依赖图进行漏洞传播分析。主要使用：

```text
反向依赖可达分析
最短路径分析
传播深度计算
组件中心性计算
组件可达性重要度计算
```

设漏洞组件为 \(c_i\)，其综合风险评分为 \(R(c_i)\)：

\[
R(c_i)=\alpha S(c_i)+\beta A(c_i)+\gamma D(c_i)+\delta P(c_i)
\]

其中传播能力评分为：

\[
P(c_i)=\lambda_1 N(c_i)+\lambda_2 L(c_i)+\lambda_3 C(c_i)+\lambda_4 I(c_i)
\]

合并后：

\[
R(c_i)=\alpha S(c_i)+\beta A(c_i)+\gamma D(c_i)+\delta[\lambda_1 N(c_i)+\lambda_2 L(c_i)+\lambda_3 C(c_i)+\lambda_4 I(c_i)]
\]

变量含义：

```text
S(c_i)：漏洞严重程度评分
A(c_i)：受影响服务数量
D(c_i)：最大传播深度
P(c_i)：漏洞传播能力评分
N(c_i)：可达服务数量
L(c_i)：最长传播路径长度
C(c_i)：组件度中心性
I(c_i)：组件可达性重要度
```

严重程度映射：

\[
S(c_i)=
\begin{cases}
4, & Critical\\
3, & High\\
2, & Medium\\
1, & Low
\end{cases}
\]

组件可达性重要度：

\[
I(c_i)=\frac{|Desc_{G^R}(c_i)|}{|V|-1}
\]

其中，\(G^R\) 为反向依赖图，\(Desc_{G^R}(c_i)\) 表示从组件 \(c_i\) 可达的后继节点集合，\(|V|\) 表示依赖图中的节点总数。

本系统采用的具体权重可写为：

\[
P(c_i)=2N(c_i)+1.5L(c_i)+10C(c_i)+20I(c_i)
\]

\[
R(c_i)=1.6S(c_i)+A(c_i)+0.8D(c_i)+0.2P(c_i)
\]

展开后：

\[
R(c_i)=1.6S(c_i)+A(c_i)+0.8D(c_i)+0.2[2N(c_i)+1.5L(c_i)+10C(c_i)+20I(c_i)]
\]

风险等级划分：

\[
Level(c_i)=
\begin{cases}
严重风险, & R(c_i)\geq 9\\
高风险, & 7\leq R(c_i)<9\\
中风险, & 4\leq R(c_i)<7\\
低风险, & R(c_i)<4
\end{cases}
\]

论文中可写：

```text
该模型综合考虑漏洞自身严重程度、受影响服务数量、传播路径深度、节点中心性和组件可达性重要度。相比只依据 CVSS 或漏洞等级的静态评估方法，该方法更能反映漏洞在跨语言项目依赖图中的实际传播风险。
```

## 九、论文写作注意点

1. 数据库写 SQLite，不要写 MySQL。
2. 登录模块可以写“基础用户管理”，不要夸大成完整权限控制系统。
3. 版本匹配算法已经增强，可以写“支持 Maven、npm、PyPI 常见版本约束”，但不要写“完全覆盖所有工业级版本规则”。
4. 风险评估公式可以作为论文重点。
5. 系统总体设计章节重在架构、流程、模块和数据库；系统详细设计章节再展开算法、表结构和模块实现。
